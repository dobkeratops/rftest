   11  /*!
   12   * A classic liveness analysis based on dataflow over the AST.  Computes,
   13   * for each local variable in a function, whether that variable is live
   14   * at a given point.  Program execution points are identified by their
   15   * id.
   16   *
   17   * # Basic idea
   18   *
   19   * The basic model is that each local variable is assigned an index.  We
   20   * represent sets of local variables using a vector indexed by this
   21   * index.  The value in the vector is either 0, indicating the variable
   22   * is dead, or the id of an expression that uses the variable.
   23   *
   24   * We conceptually walk over the AST in reverse execution order.  If we
   25   * find a use of a variable, we add it to the set of live variables.  If
   26   * we find an assignment to a variable, we remove it from the set of live
   27   * variables.  When we have to merge two flows, we take the union of
   28   * those two flows---if the variable is live on both paths, we simply
   29   * pick one id.  In the event of loops, we continue doing this until a
   30   * fixed point is reached.
   31   *
   32   * ## Checking initialization
   33   *
   34   * At the function entry point, all variables must be dead.  If this is
   35   * not the case, we can report an error using the id found in the set of
   36   * live variables, which identifies a use of the variable which is not
   37   * dominated by an assignment.
   38   *
   39   * ## Checking moves
   40   *
   41   * After each explicit move, the variable must be dead.
   42   *
   43   * ## Computing last uses
   44   *
   45   * Any use of the variable where the variable is dead afterwards is a
   46   * last use.
   47   *
   48   * # Implementation details
   49   *
   50   * The actual implementation contains two (nested) walks over the AST.
   51   * The outer walk has the job of building up the ir_maps instance for the
   52   * enclosing function.  On the way down the tree, it identifies those AST
   53   * nodes and variable IDs that will be needed for the liveness analysis
   54   * and assigns them contiguous IDs.  The liveness id for an AST node is
   55   * called a `live_node` (it's a newtype'd uint) and the id for a variable
   56   * is called a `variable` (another newtype'd uint).
   57   *
   58   * On the way back up the tree, as we are about to exit from a function
   59   * declaration we allocate a `liveness` instance.  Now that we know
   60   * precisely how many nodes and variables we need, we can allocate all
   61   * the various arrays that we will need to precisely the right size.  We then
   62   * perform the actual propagation on the `liveness` instance.
   63   *
   64   * This propagation is encoded in the various `propagate_through_*()`
   65   * methods.  It effectively does a reverse walk of the AST; whenever we
   66   * reach a loop node, we iterate until a fixed point is reached.
   67   *
   68   * ## The `Users` struct
   69   *
   70   * At each live node `N`, we track three pieces of information for each
   71   * variable `V` (these are encapsulated in the `Users` struct):
   72   *
   73   * - `reader`: the `LiveNode` ID of some node which will read the value
   74   *    that `V` holds on entry to `N`.  Formally: a node `M` such
   75   *    that there exists a path `P` from `N` to `M` where `P` does not
   76   *    write `V`.  If the `reader` is `invalid_node()`, then the current
   77   *    value will never be read (the variable is dead, essentially).
   78   *
   79   * - `writer`: the `LiveNode` ID of some node which will write the
   80   *    variable `V` and which is reachable from `N`.  Formally: a node `M`
   81   *    such that there exists a path `P` from `N` to `M` and `M` writes
   82   *    `V`.  If the `writer` is `invalid_node()`, then there is no writer
   83   *    of `V` that follows `N`.
   84   *
   85   * - `used`: a boolean value indicating whether `V` is *used*.  We
   86   *   distinguish a *read* from a *use* in that a *use* is some read that
   87   *   is not just used to generate a new value.  For example, `x += 1` is
   88   *   a read but not a use.  This is used to generate better warnings.
   89   *
   90   * ## Special Variables
   91   *
   92   * We generate various special variables for various, well, special purposes.
   93   * These are described in the `specials` struct:
   94   *
   95   * - `exit_ln`: a live node that is generated to represent every 'exit' from
   96   *   the function, whether it be by explicit return, fail, or other means.
   97   *
   98   * - `fallthrough_ln`: a live node that represents a fallthrough
   99   *
  100   * - `no_ret_var`: a synthetic variable that is only 'read' from, the
  101   *   fallthrough node.  This allows us to detect functions where we fail
  102   *   to return explicitly.
  103   */

  194  // Creating ir_maps
  195  //
  196  // This is the first pass and the one that drives the main
  197  // computation.  It walks up and down the IR once.  On the way down,
  198  // we count for each function the number of variables as well as
  199  // liveness nodes.  A liveness node is basically an expression or
  200  // capture clause that does something of interest: either it has
  201  // interesting control flow or it uses/defines a local variable.
  202  //
  203  // On the way back up, at each function node we create liveness sets
  204  // (we now know precisely how big to make our various vectors and so
  205  // forth) and then do the data-flow propagation to compute the set
  206  // of live variables at each program point.
  207  //
  208  // Finally, we run back over the IR one last time and, using the
  209  // computed liveness, check various safety conditions.  For example,
  210  // there must be no live nodes at the definition site for a variable
  211  // unless it has an initializer.  Similarly, each non-mutable local
  212  // variable must not be assigned if there is some successor
  213  // assignment.  And so forth.

  359  fn visit_fn(ir: &mut IrMaps,
  360              fk: &FnKind,
  361              decl: &FnDecl,
  362              body: &Block,
  363              sp: Span,
  364              id: NodeId) {
  365      debug!("visit_fn: id={}", id);
  366      let _i = ::util::common::indenter();
  367  
  368      // swap in a new set of IR maps for this function body:
  369      let mut fn_maps = IrMaps(ir.tcx, ir.method_map, ir.capture_map);
  370  
  371      unsafe {
  372          debug!("creating fn_maps: {}", transmute::<&IrMaps, *IrMaps>(&fn_maps));
  373      }
  374  
  375      for arg in decl.inputs.iter() {
  376          pat_util::pat_bindings(ir.tcx.def_map,
  377                                 arg.pat,
  378                                 |_bm, arg_id, _x, path| {
  379              debug!("adding argument {}", arg_id);
  380              let ident = ast_util::path_to_ident(path);
  381              fn_maps.add_variable(Arg(arg_id, ident));
  382          })
  383      };
  384  
  385      // gather up the various local variables, significant expressions,
  386      // and so forth:
  387      visit::walk_fn(&mut fn_maps, fk, decl, body, sp, id, ());
  388  
  389      // Special nodes and variables:
  390      // - exit_ln represents the end of the fn, either by return or fail
  391      // - implicit_ret_var is a pseudo-variable that represents
  392      //   an implicit return
  393      let specials = Specials {
  394          exit_ln: fn_maps.add_live_node(ExitNode),
  395          fallthrough_ln: fn_maps.add_live_node(ExitNode),
  396          no_ret_var: fn_maps.add_variable(ImplicitRet)
  397      };
  398  
  399      // compute liveness
  400      let mut lsets = Liveness(&mut fn_maps, specials);

  454  fn visit_expr(ir: &mut IrMaps, expr: &Expr) {
  455      match expr.node {
  456        // live nodes required for uses or definitions of variables:
  457        ExprPath(_) => {
  458          let def = ir.tcx.def_map.borrow().get_copy(&expr.id);
  459          debug!("expr {}: path that leads to {:?}", expr.id, def);
  460          if moves::moved_variable_node_id_from_def(def).is_some() {
  461              ir.add_live_node_for_node(expr.id, ExprNode(expr.span));
  462          }
  463          visit::walk_expr(ir, expr, ());
  464        }
  465        ExprFnBlock(..) | ExprProc(..) => {
  466          // Interesting control flow (for loops can contain labeled
  467          // breaks or continues)
  468          ir.add_live_node_for_node(expr.id, ExprNode(expr.span));
  469  
  470          // Make a live_node for each captured variable, with the span
  471          // being the location that the variable is used.  This results
  472          // in better error messages than just pointing at the closure
  473          // construction site.
  474          let mut call_caps = Vec::new();
  475          for cv in ir.capture_map.get(&expr.id).iter() {
  476              match moves::moved_variable_node_id_from_def(cv.def) {
  477                Some(rv) => {
  478                  let cv_ln = ir.add_live_node(FreeVarNode(cv.span));
  479                  let is_move = match cv.mode {
  480                      // var must be dead afterwards
  481                      moves::CapMove => true,
  482  
  483                      // var can stil be used
  484                      moves::CapCopy | moves::CapRef => false
  485                  };
  486                  call_caps.push(CaptureInfo {ln: cv_ln,
  487                                              is_move: is_move,
  488                                              var_nid: rv});
  489                }
  490                None => {}
  491              }
  492          }
  493          ir.set_captures(expr.id, call_caps);
  494  
  495          visit::walk_expr(ir, expr, ());
  496        }
  497  
  498        // live nodes required for interesting control flow:
  499        ExprIf(..) | ExprMatch(..) | ExprWhile(..) | ExprLoop(..) => {
  500          ir.add_live_node_for_node(expr.id, ExprNode(expr.span));
  501          visit::walk_expr(ir, expr, ());
  502        }
  503        ExprForLoop(..) => fail!("non-desugared expr_for_loop"),
  504        ExprBinary(op, _, _) if ast_util::lazy_binop(op) => {
  505          ir.add_live_node_for_node(expr.id, ExprNode(expr.span));
  506          visit::walk_expr(ir, expr, ());
  507        }
  508  
  509        // otherwise, live nodes are not required:
  510        ExprIndex(..) | ExprField(..) | ExprVstore(..) | ExprVec(..) |
  511        ExprCall(..) | ExprMethodCall(..) | ExprTup(..) |
  512        ExprBinary(..) | ExprAddrOf(..) |
  513        ExprCast(..) | ExprUnary(..) | ExprBreak(_) |
  514        ExprAgain(_) | ExprLit(_) | ExprRet(..) | ExprBlock(..) |
  515        ExprAssign(..) | ExprAssignOp(..) | ExprMac(..) |
  516        ExprStruct(..) | ExprRepeat(..) | ExprParen(..) |
  517        ExprInlineAsm(..) | ExprBox(..) => {
  518            visit::walk_expr(ir, expr, ());
  519        }
  520      }
  521  }

 1201      fn propagate_through_lvalue_components(&mut self,
 1202                                             expr: &Expr,
 1203                                             succ: LiveNode)
 1204                                             -> LiveNode {
 1205          // # Lvalues
 1206          //
 1207          // In general, the full flow graph structure for an
 1208          // assignment/move/etc can be handled in one of two ways,
 1209          // depending on whether what is being assigned is a "tracked
 1210          // value" or not. A tracked value is basically a local
 1211          // variable or argument.
 1212          //
 1213          // The two kinds of graphs are:
 1214          //
 1215          //    Tracked lvalue          Untracked lvalue
 1216          // ----------------------++-----------------------
 1217          //                       ||
 1218          //         |             ||           |
 1219          //         v             ||           v
 1220          //     (rvalue)          ||       (rvalue)
 1221          //         |             ||           |
 1222          //         v             ||           v
 1223          // (write of lvalue)     ||   (lvalue components)
 1224          //         |             ||           |
 1225          //         v             ||           v
 1226          //      (succ)           ||        (succ)
 1227          //                       ||
 1228          // ----------------------++-----------------------
 1229          //
 1230          // I will cover the two cases in turn:
 1231          //
 1232          // # Tracked lvalues
 1233          //
 1234          // A tracked lvalue is a local variable/argument `x`.  In
 1235          // these cases, the link_node where the write occurs is linked
 1236          // to node id of `x`.  The `write_lvalue()` routine generates
 1237          // the contents of this node.  There are no subcomponents to
