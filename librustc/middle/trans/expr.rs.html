   12 
   13 # Translation of expressions.
   14 
   15 ## Recommended entry point
   16 
   17 If you wish to translate an expression, the preferred way to do
   18 so is to use:
   19 
   20     expr::trans_into(block, expr, Dest) -> block
   21 
   22 This will generate code that evaluates `expr`, storing the result into
   23 `Dest`, which must either be the special flag ignore (throw the result
   24 away) or be a pointer to memory of the same type/size as the
   25 expression.  It returns the resulting basic block.  This form will
   26 handle all automatic adjustments for you. The value will be moved if
   27 its type is linear and copied otherwise.
   28 
   29 ## Translation to a datum
   30 
   31 In some cases, `trans_into()` is too narrow of an interface.
   32 Generally this occurs either when you know that the result value is
   33 going to be a scalar, or when you need to evaluate the expression into
   34 some memory location so you can go and inspect it (e.g., assignments,
   35 `match` expressions, the `&` operator).
   36 
   37 In such cases, you want the following function:
   38 
   39     trans_to_datum(block, expr) -> DatumBlock
   40 
   41 This function generates code to evaluate the expression and return a
   42 `Datum` describing where the result is to be found.  This function
   43 tries to return its result in the most efficient way possible, without
   44 introducing extra copies or sacrificing information.  Therefore, for
   45 lvalue expressions, you always get a by-ref `Datum` in return that
   46 points at the memory for this lvalue.  For rvalue expressions, we will
   47 return a by-value `Datum` whenever possible, but it is often necessary
   48 to allocate a stack slot, store the result of the rvalue in there, and
   49 then return a pointer to the slot (see the discussion later on about
   50 the different kinds of rvalues).
   51 
   52 NB: The `trans_to_datum()` function does perform adjustments, but
   53 since it returns a pointer to the value "in place" it does not handle
   54 moves.  If you wish to copy/move the value returned into a new
   55 location, you should use the Datum method `store_to()` (move or copy
   56 depending on type). You can also use `move_to()` (force move) or
   57 `copy_to()` (force copy) for special situations.
   58 
   59 ## Translating local variables
   60 
   61 `trans_local_var()` can be used to trans a ref to a local variable
   62 that is not an expression.  This is needed for captures.
   63 
   64 ## Ownership and cleanups
   65 
   66 The current system for cleanups associates required cleanups with
   67 block contexts.  Block contexts are structured into a tree that
   68 resembles the code itself.  Not every block context has cleanups
   69 associated with it, only those blocks that have a kind of
   70 `block_scope`.  See `common::block_kind` for more details.
   71 
   72 If you invoke `trans_into()`, no cleanup is scheduled for you.  The
   73 value is written into the given destination and is assumed to be owned
   74 by that destination.
   75 
   76 When you invoke `trans_to_datum()` on an rvalue, the resulting
   77 datum/value will have an appropriate cleanup scheduled for the
   78 innermost cleanup scope.  If you later use `move_to()` or
   79 `drop_val()`, this cleanup will be canceled.
   80 
   81 During the evaluation of an expression, temporary cleanups are created
   82 and later canceled.  These represent intermediate or partial results
   83 which must be cleaned up in the event of task failure.
   84 
   85 ## Implementation details
   86 
   87 We divide expressions into three categories, based on how they are most
   88 naturally implemented:
   89 
   90 1. Lvalues
   91 2. Datum rvalues
   92 3. DPS rvalues
   93 4. Statement rvalues
   94 
   95 Lvalues always refer to user-assignable memory locations.
   96 Translating those always results in a by-ref datum; this introduces
   97 no inefficiencies into the generated code, because all lvalues are
   98 naturally addressable.
   99 
  100 Datum rvalues are rvalues that always generate datums as a result.
  101 These are generally scalar results, such as `a+b` where `a` and `b`
  102 are integers.
  103 
  104 DPS rvalues are rvalues that, when translated, must be given a
  105 memory location to write into (or the Ignore flag).  These are
  106 generally expressions that produce structural results that are
  107 larger than one word (e.g., a struct literal), but also expressions
  108 (like `if`) that involve control flow (otherwise we'd have to
  109 generate phi nodes).
  110 
  111 Finally, statement rvalues are rvalues that always produce a nil
  112 return type, such as `while` loops or assignments (`a = b`).
  113 
  114 */

  182 pub fn trans_to_datum(bcx: @mut Block, expr: @ast::expr) -> DatumBlock {
  183     debug!("trans_to_datum(expr=%s)", bcx.expr_to_str(expr));
  184 
  185     let mut bcx = bcx;
  186     let mut datum = unpack_datum!(bcx, trans_to_datum_unadjusted(bcx, expr));
  187     let adjustment = match bcx.tcx().adjustments.find_copy(&expr.id) {
  188         None => { return DatumBlock {bcx: bcx, datum: datum}; }
  189         Some(adj) => { adj }
  190     };
  191     debug!("unadjusted datum: %s", datum.to_str(bcx.ccx()));
  192     match *adjustment {
  193         AutoAddEnv(*) => {
  194             datum = unpack_datum!(bcx, add_env(bcx, expr, datum));
  195         }
  196         AutoDerefRef(ref adj) => {
  197             if adj.autoderefs > 0 {
  198                 datum =
  199                     unpack_datum!(
  200                         bcx,
  201                         datum.autoderef(bcx, expr.span,
  202                                         expr.id, adj.autoderefs));
  203             }
  204 
  205             datum = match adj.autoref {
  206                 None => {
  207                     datum
  208                 }
  209                 Some(AutoUnsafe(*)) | // region + unsafe ptrs have same repr
  210                 Some(AutoPtr(*)) => {
  211                     unpack_datum!(bcx, auto_ref(bcx, datum))
  212                 }
  213                 Some(AutoBorrowVec(*)) => {
  214                     unpack_datum!(bcx, auto_slice(bcx, adj.autoderefs,
  215                                                   expr, datum))
  216                 }
  217                 Some(AutoBorrowVecRef(*)) => {
  218                     unpack_datum!(bcx, auto_slice_and_ref(bcx, adj.autoderefs,
  219                                                           expr, datum))
  220                 }
  221                 Some(AutoBorrowFn(*)) => {
  222                     let adjusted_ty = ty::adjust_ty(bcx.tcx(), expr.span,
  223                                                     datum.ty, Some(adjustment));
  224                     unpack_datum!(bcx, auto_borrow_fn(bcx, adjusted_ty, datum))
  225                 }
  226             };
  227         }
  228     }
  229     debug!("after adjustments, datum=%s", datum.to_str(bcx.ccx()));
  230     return DatumBlock {bcx: bcx, datum: datum};
  231

  247     fn auto_slice(bcx: @mut Block,
  248                   autoderefs: uint,
  249                   expr: &ast::expr,
  250                   datum: Datum) -> DatumBlock {
  251         // This is not the most efficient thing possible; since slices
  252         // are two words it'd be better if this were compiled in
  253         // 'dest' mode, but I can't find a nice way to structure the
  254         // code and keep it DRY that accommodates that use case at the
  255         // moment.
  256 
  257         let tcx = bcx.tcx();
  258         let unit_ty = ty::sequence_element_type(tcx, datum.ty);
  259 
  260         let (bcx, base, len) =
  261             datum.get_vec_base_and_len(bcx, expr.span, expr.id, autoderefs+1);
  262 
  263         // this type may have a different region/mutability than the
  264         // real one, but it will have the same runtime representation
  265         let slice_ty = ty::mk_evec(tcx,
  266                                    ty::mt { ty: unit_ty, mutbl: ast::m_imm },
  267                                    ty::vstore_slice(ty::re_static));
  268 
  269         let scratch = scratch_datum(bcx, slice_ty, "__adjust", false);
  270         Store(bcx, base, GEPi(bcx, scratch.val, [0u, abi::slice_elt_base]));
  271         Store(bcx, len, GEPi(bcx, scratch.val, [0u, abi::slice_elt_len]));
  272         DatumBlock {bcx: bcx, datum: scratch}
  273     }

  276         // This is not the most efficient thing possible; since closures
  277         // are two words it'd be better if this were compiled in
  278         // 'dest' mode, but I can't find a nice way to structure the
  279         // code and keep it DRY that accommodates that use case at the
  280         // moment.
  281 
  282         let tcx = bcx.tcx();
  283         let closure_ty = expr_ty_adjusted(bcx, expr);
  284         debug!("add_env(closure_ty=%s)", closure_ty.repr(tcx));
  285         let scratch = scratch_datum(bcx, closure_ty, "__adjust", false);

  289         let llenv = GEPi(bcx, scratch.val, [0u, abi::fn_field_box]);
  290         Store(bcx, base::null_env_ptr(bcx), llenv);
  291         DatumBlock {bcx: bcx, datum: scratch}
  292     }

  303 pub fn trans_into(bcx: @mut Block, expr: @ast::expr, dest: Dest) -> @mut Block {
  304     if bcx.tcx().adjustments.contains_key(&expr.id) {
  305         // use trans_to_datum, which is mildly less efficient but
  306         // which will perform the adjustments:
  307         let datumblock = trans_to_datum(bcx, expr);
  308         return match dest {
  309             Ignore => datumblock.bcx,
  310             SaveIn(lldest) => datumblock.store_to(INIT, lldest)
  311         };
  312     }
  313 
  314     let ty = expr_ty(bcx, expr);
  315 
  316     debug!("trans_into_unadjusted(expr=%s, dest=%s)",
  317            bcx.expr_to_str(expr),
  318            dest.to_str(bcx.ccx()));
  319     let _indenter = indenter();
  320 
  321     debuginfo::update_source_pos(bcx, expr.span);
  322 
  323     let dest = {
  324         if ty::type_is_nil(ty) || ty::type_is_bot(ty) {
  325             Ignore
  326         } else {
  327             dest
  328         }
  329     };
  330 
  331     let kind = bcx.expr_kind(expr);
  332     debug!("expr kind = %?", kind);
  333     return match kind {
  334         ty::LvalueExpr => {
  335             let datumblock = trans_lvalue_unadjusted(bcx, expr);
  336             match dest {
  337                 Ignore => datumblock.bcx,
  338                 SaveIn(lldest) => datumblock.store_to(INIT, lldest)
  339             }
  340         }
  341         ty::RvalueDatumExpr => {
  342             let datumblock = trans_rvalue_datum_unadjusted(bcx, expr);
  343             match dest {
  344                 Ignore => datumblock.drop_val(),
  345 
  346                 // When processing an rvalue, the value will be newly
  347                 // allocated, so we always `move_to` so as not to
  348                 // unnecessarily inc ref counts and so forth:
  349                 SaveIn(lldest) => datumblock.move_to(INIT, lldest)
  350             }
  351         }
  352         ty::RvalueDpsExpr => {
  353             trans_rvalue_dps_unadjusted(bcx, expr, dest)
  354         }
  355         ty::RvalueStmtExpr => {
  356             trans_rvalue_stmt_unadjusted(bcx, expr)
  357         }
  358     };
  359 }

  380 fn trans_to_datum_unadjusted(bcx: @mut Block, expr: @ast::expr) -> DatumBlock {
  381     /*!
  382      * Translates an expression into a datum.  If this expression
  383      * is an rvalue, this will result in a temporary value being
  384      * created.  If you plan to store the value somewhere else,
  385      * you should prefer `trans_into()` instead.
  386      */
  387 
  388     let mut bcx = bcx;
  389 
  390     debug!("trans_to_datum_unadjusted(expr=%s)", bcx.expr_to_str(expr));
  391     let _indenter = indenter();
  392 
  393     debuginfo::update_source_pos(bcx, expr.span);
  394 
  395     match ty::expr_kind(bcx.tcx(), bcx.ccx().maps.method_map, expr) {
  396         ty::LvalueExpr => {
  397             return trans_lvalue_unadjusted(bcx, expr);
  398         }
  399 
  400         ty::RvalueDatumExpr => {
  401             let datum = unpack_datum!(bcx, {
  402                 trans_rvalue_datum_unadjusted(bcx, expr)
  403             });
  404             datum.add_clean(bcx);
  405             return DatumBlock {bcx: bcx, datum: datum};
  406         }
  407 
  408         ty::RvalueStmtExpr => {
  409             bcx = trans_rvalue_stmt_unadjusted(bcx, expr);
  410             return nil(bcx, expr_ty(bcx, expr));
  411         }
  412 
  413         ty::RvalueDpsExpr => {
  414             let ty = expr_ty(bcx, expr);
  415             if ty::type_is_nil(ty) || ty::type_is_bot(ty) {
  416                 bcx = trans_rvalue_dps_unadjusted(bcx, expr, Ignore);
  417                 return nil(bcx, ty);
  418             } else {
  419                 let scratch = scratch_datum(bcx, ty, "", false);
  420                 bcx = trans_rvalue_dps_unadjusted(
  421                     bcx, expr, SaveIn(scratch.val));
  422 
  423                 // Note: this is not obviously a good idea.  It causes
  424                 // immediate values to be loaded immediately after a
  425                 // return from a call or other similar expression,
  426                 // which in turn leads to alloca's having shorter
  427                 // lifetimes and hence larger stack frames.  However,
  428                 // in turn it can lead to more register pressure.
  429                 // Still, in practice it seems to increase
  430                 // performance, since we have fewer problems with
  431                 // morestack churn.
  432                 let scratch = scratch.to_appropriate_datum(bcx);
  433 
  434                 scratch.add_clean(bcx);
  435                 return DatumBlock {bcx: bcx, datum: scratch};
  436             }
  437         }
  438     }
