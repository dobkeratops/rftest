  13 # Type inference engine
  14 
  15 This is loosely based on standard HM-type inference, but with an
  16 extension to try and accommodate subtyping.  There is nothing
  17 principled about this extension; it's sound---I hope!---but it's a
  18 heuristic, ultimately, and does not guarantee that it finds a valid
  19 typing even if one exists (in fact, there are known scenarios where it
  20 fails, some of which may eventually become problematic).
  21 
  22 ## Key idea
  23 
  24 The main change is that each type variable T is associated with a
  25 lower-bound L and an upper-bound U.  L and U begin as bottom and top,
  26 respectively, but gradually narrow in response to new constraints
  27 being introduced.  When a variable is finally resolved to a concrete
  28 type, it can (theoretically) select any type that is a supertype of L
  29 and a subtype of U.
  30 
  31 There are several critical invariants which we maintain:
  32 
  33 - the upper-bound of a variable only becomes lower and the lower-bound
  34   only becomes higher over time;
  35 - the lower-bound L is always a subtype of the upper bound U;
  36 - the lower-bound L and upper-bound U never refer to other type variables,
  37   but only to types (though those types may contain type variables).
  38 
  39 > An aside: if the terms upper- and lower-bound confuse you, think of
  40 > "supertype" and "subtype".  The upper-bound is a "supertype"
  41 > (super=upper in Latin, or something like that anyway) and the lower-bound
  42 > is a "subtype" (sub=lower in Latin).  I find it helps to visualize
  43 > a simple class hierarchy, like Java minus interfaces and
  44 > primitive types.  The class Object is at the root (top) and other
  45 > types lie in between.  The bottom type is then the Null type.
  46 > So the tree looks like:
  47 >
  48 >             Object
  49 >             /    \
  50 >         String   Other
  51 >             \    /
  52 >             (null)
  53 >
  54 > So the upper bound type is the "supertype" and the lower bound is the
  55 > "subtype" (also, super and sub mean upper and lower in Latin, or something
  56 > like that anyway).
  57 
  58 ## Satisfying constraints
  59 
  60 At a primitive level, there is only one form of constraint that the
  61 inference understands: a subtype relation.  So the outside world can
  62 say "make type A a subtype of type B".  If there are variables
  63 involved, the inferencer will adjust their upper- and lower-bounds as
  64 needed to ensure that this relation is satisfied. (We also allow "make
  65 type A equal to type B", but this is translated into "A <: B" and "B
  66 <: A")
  67 
  68 As stated above, we always maintain the invariant that type bounds
  69 never refer to other variables.  This keeps the inference relatively
  70 simple, avoiding the scenario of having a kind of graph where we have
  71 to pump constraints along and reach a fixed point, but it does impose
  72 some heuristics in the case where the user is relating two type
  73 variables A <: B.
  74 
  75 Combining two variables such that variable A will forever be a subtype
  76 of variable B is the trickiest part of the algorithm because there is
  77 often no right choice---that is, the right choice will depend on
  78 future constraints which we do not yet know. The problem comes about
  79 because both A and B have bounds that can be adjusted in the future.
  80 Let's look at some of the cases that can come up.
  81 
  82 Imagine, to start, the best case, where both A and B have an upper and
  83 lower bound (that is, the bounds are not top nor bot respectively). In
  84 that case, if we're lucky, A.ub <: B.lb, and so we know that whatever
  85 A and B should become, they will forever have the desired subtyping
  86 relation.  We can just leave things as they are.
  87 
  88 ### Option 1: Unify
  89 
  90 However, suppose that A.ub is *not* a subtype of B.lb.  In
  91 that case, we must make a decision.  One option is to unify A
  92 and B so that they are one variable whose bounds are:
  93 
  94     UB = GLB(A.ub, B.ub)
  95     LB = LUB(A.lb, B.lb)
  96 
  97 (Note that we will have to verify that LB <: UB; if it does not, the
  98 types are not intersecting and there is an error) In that case, A <: B
  99 holds trivially because A==B.  However, we have now lost some
 100 flexibility, because perhaps the user intended for A and B to end up
 101 as different types and not the same type.
 102 
 103 Pictorally, what this does is to take two distinct variables with
 104 (hopefully not completely) distinct type ranges and produce one with
 105 the intersection.
 106 
 107                       B.ub                  B.ub
 108                        /\                    /
 109                A.ub   /  \           A.ub   /
 110                /   \ /    \              \ /
 111               /     X      \              UB
 112              /     / \      \            / \
 113             /     /   /      \          /   /
 114             \     \  /       /          \  /
 115              \      X       /             LB
 116               \    / \     /             / \
 117                \  /   \   /             /   \
 118                A.lb    B.lb          A.lb    B.lb
 119 
 120 
 121 ### Option 2: Relate UB/LB
 122 
 123 Another option is to keep A and B as distinct variables but set their
 124 bounds in such a way that, whatever happens, we know that A <: B will hold.
 125 This can be achieved by ensuring that A.ub <: B.lb.  In practice there
 126 are two ways to do that, depicted pictorally here:
 127 
 128         Before                Option #1            Option #2
 129 
 130                  B.ub                B.ub                B.ub
 131                   /\                 /  \                /  \
 132           A.ub   /  \        A.ub   /(B')\       A.ub   /(B')\
 133           /   \ /    \           \ /     /           \ /     /
 134          /     X      \         __UB____/             UB    /
 135         /     / \      \       /  |                   |    /
 136        /     /   /      \     /   |                   |   /
 137        \     \  /       /    /(A')|                   |  /
 138         \      X       /    /     LB            ______LB/
 139          \    / \     /    /     / \           / (A')/ \
 140           \  /   \   /     \    /   \          \    /   \
 141           A.lb    B.lb       A.lb    B.lb        A.lb    B.lb
 142 
 143 In these diagrams, UB and LB are defined as before.  As you can see,
 144 the new ranges `A'` and `B'` are quite different from the range that
 145 would be produced by unifying the variables.
 146 
 147 ### What we do now
 148 
 149 Our current technique is to *try* (transactionally) to relate the
 150 existing bounds of A and B, if there are any (i.e., if `UB(A) != top
 151 && LB(B) != bot`).  If that succeeds, we're done.  If it fails, then
 152 we merge A and B into same variable.
 153 
 154 This is not clearly the correct course.  For example, if `UB(A) !=
 155 top` but `LB(B) == bot`, we could conceivably set `LB(B)` to `UB(A)`
 156 and leave the variables unmerged.  This is sometimes the better
 157 course, it depends on the program.
 158 
 159 The main case which fails today that I would like to support is:
 160 
 161     fn foo<T>(x: T, y: T) { ... }
 162 
 163     fn bar() {
 164         let x: @mut int = @mut 3;
 165         let y: @int = @3;
 166         foo(x, y);
 167     }
 168 
 169 In principle, the inferencer ought to find that the parameter `T` to
 170 `foo(x, y)` is `@const int`.  Today, however, it does not; this is
 171 because the type variable `T` is merged with the type variable for
 172 `X`, and thus inherits its UB/LB of `@mut int`.  This leaves no
 173 flexibility for `T` to later adjust to accommodate `@int`.
 174 
 175 ### What to do when not all bounds are present
 176 
 177 In the prior discussion we assumed that A.ub was not top and B.lb was
 178 not bot.  Unfortunately this is rarely the case.  Often type variables
 179 have "lopsided" bounds.  For example, if a variable in the program has
 180 been initialized but has not been used, then its corresponding type
 181 variable will have a lower bound but no upper bound.  When that
 182 variable is then used, we would like to know its upper bound---but we
 183 don't have one!  In this case we'll do different things depending on
 184 how the variable is being used.
 185 
 186 ## Transactional support
 187 
 188 Whenever we adjust merge variables or adjust their bounds, we always
 189 keep a record of the old value.  This allows the changes to be undone.
 190 
 191 ## Regions
 192 
 193 I've only talked about type variables here, but region variables
 194 follow the same principle.  They have upper- and lower-bounds.  A
 195 region A is a subregion of a region B if A being valid implies that B
 196 is valid.  This basically corresponds to the block nesting structure:
 197 the regions for outer block scopes are superregions of those for inner
 198 block scopes.
 199 
 200 ## Integral and floating-point type variables
 201 
 202 There is a third variety of type variable that we use only for
 203 inferring the types of unsuffixed integer literals.  Integral type
 204 variables differ from general-purpose type variables in that there's
 205 no subtyping relationship among the various integral types, so instead
 206 of associating each variable with an upper and lower bound, we just
 207 use simple unification.  Each integer variable is associated with at
 208 most one integer type.  Floating point types are handled similarly to
 209 integral types.
 210 
 211 ## GLB/LUB
 212 
 213 Computing the greatest-lower-bound and least-upper-bound of two
 214 types/regions is generally straightforward except when type variables
 215 are involved. In that case, we follow a similar "try to use the bounds
 216 when possible but otherwise merge the variables" strategy.  In other
 217 words, `GLB(A, B)` where `A` and `B` are variables will often result
 218 in `A` and `B` being merged and the result being `A`.
 219 
 220 ## Type coercion
 221 
 222 We have a notion of assignability which differs somewhat from
 223 subtyping; in particular it may cause region borrowing to occur.  See
 224 the big comment later in this file on Type Coercion for specifics.
 225 
 226 ### In conclusion
 227 
 228 I showed you three ways to relate `A` and `B`.  There are also more,
 229 of course, though I'm not sure if there are any more sensible options.
 230 The main point is that there are various options, each of which
 231 produce a distinct range of types for `A` and `B`.  Depending on what
 232 the correct values for A and B are, one of these options will be the
 233 right choice: but of course we don't know the right values for A and B
 234 yet, that's what we're trying to find!  In our code, we opt to unify
 235 (Option #1).

 237 # Implementation details
 238 
 239 We make use of a trait-like impementation strategy to consolidate
 240 duplicated code between subtypes, GLB, and LUB computations.  See the
 241 section on "Type Combining" below for details.
