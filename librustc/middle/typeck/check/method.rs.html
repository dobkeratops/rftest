   12  
   13  # Method lookup
   14  
   15  Method lookup can be rather complex due to the interaction of a number
   16  of factors, such as self types, autoderef, trait lookup, etc.  The
   17  algorithm is divided into two parts: candidate collection and
   18  candidate selection.
   19  
   20  ## Candidate collection
   21  
   22  A `Candidate` is a method item that might plausibly be the method
   23  being invoked.  Candidates are grouped into two kinds, inherent and
   24  extension.  Inherent candidates are those that are derived from the
   25  type of the receiver itself.  So, if you have a receiver of some
   26  nominal type `Foo` (e.g., a struct), any methods defined within an
   27  impl like `impl Foo` are inherent methods.  Nothing needs to be
   28  imported to use an inherent method, they are associated with the type
   29  itself (note that inherent impls can only be defined in the same
   30  module as the type itself).
   31  
   32  Inherent candidates are not always derived from impls.  If you have a
   33  trait instance, such as a value of type `@ToStr`, then the trait
   34  methods (`to_str()`, in this case) are inherently associated with it.
   35  Another case is type parameters, in which case the methods of their
   36  bounds are inherent.
   37  
   38  Extension candidates are derived from imported traits.  If I have the
   39  trait `ToStr` imported, and I call `to_str()` on a value of type `T`,
   40  then we will go off to find out whether there is an impl of `ToStr`
   41  for `T`.  These kinds of method calls are called "extension methods".
   42  They can be defined in any module, not only the one that defined `T`.
   43  Furthermore, you must import the trait to call such a method.
   44  
   45  For better or worse, we currently give weight to inherent methods over
   46  extension methods during candidate selection (below).
   47  
   48  ## Candidate selection
   49  
   50  Once we know the set of candidates, we can go off and try to select
   51  which one is actually being called.  We do this by taking the type of
   52  the receiver, let's call it R, and checking whether it matches against
   53  the expected receiver type for each of the collected candidates.  We
   54  first check for inherent candidates and see whether we get exactly one
   55  match (zero means keep searching, more than one is an error).  If so,
   56  we return that as the candidate.  Otherwise we search the extension
   57  candidates in the same way.
   58  
   59  If find no matching candidate at all, we proceed to auto-deref the
   60  receiver type and search again.  We keep doing that until we cannot
   61  auto-deref any longer.  At each step, we also check for candidates
   62  based on "autoptr", which if the current type is `T`, checks for `&mut
   63  T`, `&const T`, and `&T` receivers.  Finally, at the very end, we will
   64  also try autoslice, which converts `~[]` to `&[]` (there is no point
   65  at trying autoslice earlier, because no autoderefable type is also
   66  sliceable).
   67  
   68  ## Why two phases?
   69  
   70  You might wonder why we first collect the candidates and then select.
   71  Both the inherent candidate collection and the candidate selection
   72  proceed by progressively deref'ing the receiver type, after all.  The
   73  answer is that two phases are needed to elegantly deal with explicit
   74  self.  After all, if there is an impl for the type `Foo`, it can
   75  define a method with the type `~self`, which means that it expects a
   76  receiver of type `~Foo`.  If we have a receiver of type `~Foo`, but we
   77  waited to search for that impl until we have deref'd the `~` away and
   78  obtained the type `Foo`, we would never match this method.
   79  
   80  */

  150             self_expr.repr(fcx.tcx()));

  162      lcx.push_extension_candidates(expr.id);
  163      lcx.search(self_ty)
  164  }

  203  // to a trait and its supertraits.
  204  fn get_method_index(tcx: &ty::ctxt,
  205                      trait_ref: &TraitRef,
  206                      subtrait: @TraitRef,
  207                      n_method: uint) -> uint {
  208      // We need to figure the "real index" of the method in a
  209      // listing of all the methods of an object. We do this by
  210      // iterating down the supertraits of the object's trait until
  211      // we find the trait the method came from, counting up the
  212      // methods from them.
  213      let mut method_count = 0;
  214      ty::each_bound_trait_and_supertraits(tcx, &[subtrait], |bound_ref| {
  215          if bound_ref.def_id == trait_ref.def_id { false }
  216              else {
  217              method_count += ty::trait_methods(tcx, bound_ref.def_id).len();
  218              true
  219          }
  220      });
  221      method_count + n_method
  222  }

  224  fn construct_transformed_self_ty_for_object(
  225      tcx: &ty::ctxt,
  226      span: Span,
  227      trait_def_id: ast::DefId,
  228      rcvr_substs: &ty::substs,
  229      method_ty: &ty::Method)
  230      -> ty::t {
  231      /*!
  232          * This is a bit tricky. We have a match against a trait method
  233          * being invoked on an object, and we want to generate the
  234          * self-type. As an example, consider a trait
  235          *
  236          *     trait Foo {
  237          *         fn r_method<'a>(&'a self);
  238          *         fn u_method(~self);
  239          *     }
  240          *
  241          * Now, assuming that `r_method` is being called, we want the
  242          * result to be `&'a Foo`. Assuming that `u_method` is being
  243          * called, we want the result to be `~Foo`. Of course,
  244          * this transformation has already been done as part of
  245          * `method_ty.fty.sig.inputs[0]`, but there the type
  246          * is expressed in terms of `Self` (i.e., `&'a Self`, `~Self`).
  247          * Because objects are not standalone types, we can't just substitute
  248          * `s/Self/Foo/`, so we must instead perform this kind of hokey
  249          * match below.
  250          */
  251  
  252      let substs = ty::substs {regions: rcvr_substs.regions.clone(),
  253                                  self_ty: None,
  254                                  tps: rcvr_substs.tps.clone()};
  255      match method_ty.explicit_self {
  256          ast::SelfStatic => {
  257              tcx.sess.span_bug(span, "static method for object type receiver");
  258          }
  259          ast::SelfValue => {
  260              ty::mk_err() // error reported in `enforce_object_limitations()`
  261          }
  262          ast::SelfRegion(..) | ast::SelfUniq => {
  263              let transformed_self_ty = *method_ty.fty.sig.inputs.get(0);
  264              match ty::get(transformed_self_ty).sty {
  265                  ty::ty_rptr(r, mt) => { // must be SelfRegion
  266                      ty::mk_trait(tcx, trait_def_id, substs,
  267                                   RegionTraitStore(r), mt.mutbl,
  268                                   ty::EmptyBuiltinBounds())
  269                  }
  270                  ty::ty_uniq(_) => { // must be SelfUniq
  271                      ty::mk_trait(tcx, trait_def_id, substs,
  272                                   UniqTraitStore, ast::MutImmutable,
  273                                   ty::EmptyBuiltinBounds())
  274                  }
  275                  _ => {
  276                      tcx.sess.span_bug(span,
  277                          format!("'impossible' transformed_self_ty: {}",
  278                                  transformed_self_ty.repr(tcx)));
  279                  }
  280              }
  281          }
  282      }
  283  }

  317  /// This type represents the conditions under which the receiver is
  318  /// considered to "match" a given method candidate. Typically the test
  319  /// is whether the receiver is of a particular type. However, this
  320  /// type is the type of the receiver *after accounting for the
  321  /// method's self type* (e.g., if the method is an `~self` method, we
  322  /// have *already verified* that the receiver is of some type `~T` and
  323  /// now we must check that the type `T` is correct).  Unfortunately,
  324  /// because traits are not types, this is a pain to do.
  325  #[deriving(Clone)]
  326  pub enum RcvrMatchCondition {
  327      RcvrMatchesIfObject(ast::DefId),
  328      RcvrMatchesIfSubtype(ty::t)
  329  }

  331  impl<'a> LookupContext<'a> {
  332      fn search(&self, self_ty: ty::t) -> Option<MethodCallee> {
  333          let span = self.self_expr.map_or(self.span, |e| e.span);
  334          let self_expr_id = self.self_expr.map(|e| e.id);
  335          let (self_ty, autoderefs, result) =
  336              check::autoderef(
  337                  self.fcx, span, self_ty, self_expr_id, PreferMutLvalue,
  338                  |self_ty, autoderefs| self.search_step(self_ty, autoderefs));
  339  
  340          match result {
  341              Some(Some(result)) => Some(result),
  342              _ => {
  343                  if self.is_overloaded_deref() {
  344                      // If we are searching for an overloaded deref, no
  345                      // need to try coercing a `~[T]` to an `&[T]` and
  346                      // searching for an overloaded deref on *that*.
  347                      None
  348                  } else {
  349                      self.search_for_autosliced_method(self_ty, autoderefs)
  350                  }
  351              }
  352          }
  353      }

  357                     autoderefs: uint)
  358                     -> Option<Option<MethodCallee>> {

  360                 self.ty_to_str(self_ty), autoderefs);

  410           * `self.inherent_candidates`.  See comment at the start of
  411           * the file.  To find the inherent candidates, we repeatedly
  412           * deref the self-ty to find the "base-type".  So, for
  413           * example, if the receiver is @@C where `C` is a struct type,
  414           * we'll want to find the inherent impls for `C`.

  651      fn push_candidates_from_impl(&mut self,
  652                                   impl_info: &ty::Impl,
  653                                   is_extension: bool) {

  655              return; // already visited

  680              substs: impl_substs,

  724                          self.fcx.write_adjustment(self_expr_id, adj);

  733      fn consider_reborrow(&self,
  734                           self_ty: ty::t,
  735                           autoderefs: uint)
  736                           -> (ty::t, ty::AutoDerefRef) {
  737          /*!
  738           * In the event that we are invoking a method with a receiver
  739           * of a borrowed type like `&T`, `&mut T`, or `&mut [T]`,
  740           * we will "reborrow" the receiver implicitly.  For example, if
  741           * you have a call `r.inc()` and where `r` has type `&mut T`,
  742           * then we treat that like `(&mut *r).inc()`.  This avoids
  743           * consuming the original pointer.
  744           *
  745           * You might think that this would be a natural byproduct of
  746           * the auto-deref/auto-ref process.  This is true for `~T`
  747           * but not for an `&mut T` receiver.  With `~T`, we would
  748           * begin by testing for methods with a self type `~T`,
  749           * then autoderef to `T`, then autoref to `&mut T`.  But with
  750           * an `&mut T` receiver the process begins with `&mut T`, only
  751           * without any autoadjustments.
  752           */
  753  
  754          let tcx = self.tcx();
  755          return match ty::get(self_ty).sty {
  756              ty::ty_rptr(_, self_mt) if default_method_hack(self_mt) => {
  757                  (self_ty,
  758                   ty::AutoDerefRef {
  759                       autoderefs: autoderefs,
  760                       autoref: None})
  761              }
  762              ty::ty_rptr(_, self_mt) => {
  763                  let region =
  764                      self.infcx().next_region_var(infer::Autoref(self.span));
  765                  (ty::mk_rptr(tcx, region, self_mt),
  766                   ty::AutoDerefRef {
  767                       autoderefs: autoderefs+1,
  768                       autoref: Some(ty::AutoPtr(region, self_mt.mutbl))})
  769              }
  770              ty::ty_vec(self_mt, vstore_slice(_)) => {
  771                  let region =
  772                      self.infcx().next_region_var(infer::Autoref(self.span));
  773                  (ty::mk_vec(tcx, self_mt, vstore_slice(region)),
  774                   ty::AutoDerefRef {
  775                       autoderefs: autoderefs,
  776                       autoref: Some(ty::AutoBorrowVec(region, self_mt.mutbl))})
  777              }
  778              ty::ty_trait(~ty::TyTrait {
  779                  def_id, ref substs, store: ty::RegionTraitStore(_), mutability: mutbl, bounds
  780              }) => {
  781                  let region =
  782                      self.infcx().next_region_var(infer::Autoref(self.span));
  783                  (ty::mk_trait(tcx, def_id, substs.clone(),
  784                                ty::RegionTraitStore(region),
  785                                mutbl, bounds),
  786                   ty::AutoDerefRef {
  787                       autoderefs: autoderefs,
  788                       autoref: Some(ty::AutoBorrowObj(region, mutbl))})
  789              }
  790              _ => {
  791                  (self_ty,
  792                   ty::AutoDerefRef {
  793                       autoderefs: autoderefs,
  794                       autoref: None})
  795              }
  796          };

 1059      fn confirm_candidate(&self, rcvr_ty: ty::t, candidate: &Candidate)

 1062          // 1. Substitute values for any type/lifetime parameters from the impl and
 1063          //    method declaration into the method type. This is the function type
 1064          //    before it is called; it may still include late bound region variables.

 1066          //    with fresh region variables.
 1067  
 1068          let tcx = self.tcx();

 1071                 self.ty_to_str(rcvr_ty),
 1072                 candidate.repr(self.tcx()));
 1073  
 1074          self.enforce_object_limitations(candidate);
 1075          self.enforce_drop_trait_limitations(candidate);
 1076  
 1077          // static methods should never have gotten this far:
 1078          assert!(candidate.method_ty.explicit_self != SelfStatic);
 1079  
 1080          // Determine the values for the generic parameters of the method.
 1081          // If they were not explicitly supplied, just construct fresh
 1082          // variables.
 1083          let num_supplied_tps = self.supplied_tps.len();
 1084          let num_method_tps = candidate.method_ty.generics.type_param_defs().len();

 1087                  self.fcx.infcx().next_ty_vars(num_method_tps)

 1090                      self.span,
 1091                      "this method does not take type parameters");
 1092                  self.fcx.infcx().next_ty_vars(num_method_tps)

 1095                      self.span,
